## ⚠️ 주의해야 할 작업 패턴 (시작 **전** 체크)

### 패턴 1 — 긴 사실 리스트 생성

**신호어**: "모두 나열하라", "1번부터 N번까지", "전체 목록", "역사적으로 ~한 사람들", "~의 종류를 다 적어"

**위험도**: 🔴 매우 높음

**증거**: #12 조선 27대 왕 — 13대를 "현종"으로 환각, 광해군/명종/효종 누락, "분량상 생략"으로 회피. 시스템 프롬프트의 "모르면 모른다"를 **명시적으로 어김**.

**규칙**:
- 5~10개를 넘는 순차적 사실 회상은 **거의 확실히 환각**이 섞인다.
- "모른다고 하라"는 시스템 프롬프트는 이 압력 앞에서 못 버틴다.
- **외부 DB/API에서 검증된 리스트를 먼저 가져온 뒤 LLM은 거기서 골라내거나 요약하는 역할만** 시킨다.

---

### 패턴 2 — 정의/전제를 의심해야 풀리는 문제

**신호어**: 함정문제, 수수께끼, "당연해 보이지만", "관습적으로", 농담 같은 형식

**위험도**: 🔴 높음

**증거**: #6 "방 안의 살인자" — 모델이 "들어와 죽인 사람도 살인자"라는 **정의 확장**을 못 함. 시스템 프롬프트의 "모르는 사실 추측 금지"를 너무 모범적으로 따라가다 함정 회피.

**규칙**:
- 모델은 "이 문제에 함정이 있을 수 있다"는 메타 의심을 디폴트로 안 켠다.
- **프롬프트에 "이 문제는 단어의 정의를 의심해야 풀린다"라고 한 줄만 추가해도** 정답률 오름.
- 비교: #7 외과의사 → 사회적 편견 깨기는 학습 데이터에 많아 통과. 즉 **"본 적 있는 트릭"은 풀고 "정의 비틀기"는 못 푼다**.

---

### 패턴 3 — 다단계·상호작용 단서 결합 추리

**신호어**: "다음 단서들을 종합하여", "가장 의심스러운 사람", "원인을 추론하라", 여러 증거+진술 조합

**위험도**: 🟡 중간

**증거**: #18 USB 사건 — 모델이 "물리 증거(R 책상)" → "R 의심"으로 **선형 가중치 결합**. 함정 단서 "서랍이 안 잠겨있다 = 누구나 넣을 수 있음 = 모함 가능"을 못 봄. 단서가 서로를 **무력화**하는 메타 관계는 못 잡는다.

**규칙**:
- 각 단서를 독립적으로 점수 매기는 건 잘함. **단서 간 상호작용**(부정/조건/메타)은 약함.
- 답을 받은 뒤 "이 답을 무력화할 단서가 있는가?"라고 별도 세션에서 verifier 돌릴 것.

---

### 패턴 4 — 답이 여럿 가능한 모호 문제

**신호어**: 조건이 부족한 논리 퍼즐, 해석에 따라 답이 갈리는 문제, 정책 판단

**위험도**: 🟡 중간

**증거**: #1 진술 퍼즐 — 사실 정답이 **A 또는 C 둘 다 가능**(둘 다 진술 한 개만 참 조건 만족)인데, 이전 세션에선 "C로 단정", 이번 세션에선 "둘 다 가능하지만 보통 C"로 답이 다름.

**규칙**:
- 같은 입력에 다른 답이 나올 수 있다 = **자기 일관성(self-consistency) 부족**.
- 중요 결정에 쓰면 N번(3~5번) 샘플링 후 다수결 또는 답 분포 보고 사람이 결정.

답을 받았을 때 아래 중 하나라도 보이면 **재검증 필수**:

- [ ] "일반적으로", "보통", "통상적으로" 뒤에 구체적 사실 — 출처 없음
- [ ] 같은 항목이 두 번 나옴 (#12에서 철종이 18대, 22대 두 번 나옴)

- [ ] 시스템 프롬프트의 "모른다고 하라"를 어기고 추측 시작

→ 하나라도 걸리면: 외부 검증 / 다른 모델로 재질의 / 사람 확인.

## 🛡️ 권장 하네스 (위험 작업을 어느 정도 끌어올리는 법)

### A. Self-consistency (모호 문제용)
같은 질문 N=3~5회 반복 → 다수결 답 채택. 답 분포 자체도 본다(2:3이면 사람 개입).
→ **#1 같은 모호 문제에 효과적**.

### B. Verifier 분리 (함정/추리용)
- Pass 1: 풀이자가 답을 낸다.
- Pass 2: **별도 세션**에서 "이 답에 함정/모순/누락된 단서가 있나? 반박해라"로 비판.
- Pass 3: Pass 1이 비판을 보고 최종 답 수정.
→ **#6, #18에 효과적**. 같은 모델이라도 분리 세션이면 다른 사고 경로 탐색 가능.

### C. 페르소나 앙상블 (다양한 가설 필요한 작업)
같은 문제를 "수학자/탐정/회의주의자/초딩" 등 다른 페르소나로 각각 풀게 한 뒤 심판이 종합.
→ 다양성을 sampling이 아닌 **conditioning** 레벨에서 만듬.

### D. 외부 검증 호출 (사실 환각 위험)
- 사실 리스트 → DB/API 사전 조회 후 LLM은 검증된 리스트만 다룸
- 수학 → sympy, numpy로 후검증
- 코드 → 실제 실행
→ **#12 같은 환각 사고 원천 차단**.
